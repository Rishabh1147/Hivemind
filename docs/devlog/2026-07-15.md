# Devlog — 2026-07-15

## What got built

First real slice of TriageBot: a ticket now gets a genuine Claude-backed classification, synchronously, end to end. No Kafka yet — that's a deliberate scope cut, see below.

- **LangChain4j + Anthropic wired in** — `pom.xml` imports `langchain4j-bom` (1.17.2) and adds `langchain4j-anthropic`. `infra/config/ClaudeConfig.java:1` exposes a single `ChatModel` bean built from `AnthropicChatModel`, configured from `application.yml` (`hivemind.llm.model`, `.api-key`, `.max-tokens`). Model is `claude-sonnet-5` — updated from the Day-1 placeholder `claude-sonnet-4-6`.
- **`platform/llm/LlmClient.java:1`** — thin wrapper around LangChain4j's `ChatModel`. Every vertical talks to Claude through this, never through LangChain4j types directly, so cost tracking (`CostTracker` in the roadmap) and OTel spans can be added here later without touching agent code.
- **`platform/agent/` core** — `AgentContext` (request-scoped attribute bag agents read/write to hand off state), `AgentResult<T>` (success/failure + payload, generic over any agent's output type), `BaseAgent<T>` (abstract `handle(AgentContext)`). This is the minimum shape needed to add a second agent later without redesigning anything.
- **First real agent: `verticals/triage/agents/ClassifierAgent.java:1`** — annotated `@AgentRole(vertical="triage", role="classifier")`. Sends the ticket body to Claude with a system prompt demanding strict JSON (`{"category": ..., "confidence": ...}`), parses with Jackson into a `Category` enum (`BILLING/BUG/FEATURE_REQUEST/ABUSE/OTHER`) + confidence score.
- **`TriageController.java:1`** — no longer a stub. `POST /api/v1/triage/tickets` builds an `AgentContext`, invokes `ClassifierAgent` synchronously, and returns the real classification (or a structured failure) instead of a random UUID.

## Why synchronous, no Kafka yet

The resume draft (see [[hivemind-workflow]]) claims "plan, dispatch, and coordinate subtasks via a Kafka event bus" — that's not true yet, and won't be until the event bus infra (broker, topic config, consumer wiring) exists. Building the LLM-calling agent first and wiring it through Kafka next session, rather than faking both at once, means every devlog entry only claims what's actually running.

## A real bug found by actually running it

Wrote the code, then ran the app with a dummy `ANTHROPIC_API_KEY` and hit the endpoint for real (`curl -X POST .../tickets`) instead of trusting the mocked unit tests alone. Got a raw 500 instead of a clean error response. Root cause: `ClassifierAgent.handle()` only wrapped the *JSON-parsing* step in try/catch — the `llmClient.complete(...)` call itself was outside it, so the real `AuthenticationException` LangChain4j threw (confirmed via server log: `invalid x-api-key`) propagated unhandled straight to Spring's default error handler.

Fixed by wrapping the LLM call in its own try/catch, returning `AgentResult.failure(...)` instead of letting it bubble. Added a test (`ClassifierAgentTest.failsGracefullyWhenLlmCallThrows`) that asserts this with a mocked throwing `LlmClient`, so this can't silently regress.

**Lesson for interview conversations:** mocked unit tests only prove the code you wrote does what you told it to — they can't catch "I forgot to wrap this specific line in error handling." That only showed up by actually invoking the running service.

## Known simplification (not a bug, a scope decision)

Both LLM-call failures (auth, rate limit, network) and JSON-parse failures currently return the same generic `422 Unprocessable Entity` from the controller. Distinguishing "upstream provider failed, maybe retry" from "the model returned garbage, don't retry" needs a proper error-type on `AgentResult`, which is the natural job of the `ToolInvoker` (timeout/retry/sandbox) from the roadmap — not something to bolt on ad hoc today.

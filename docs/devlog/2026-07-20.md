# Devlog — 2026-07-20

## What got built

The `ToolRegistry`/`ToolInvoker` — flagged as a candidate in both the 2026-07-16 and 2026-07-17
devlogs, built today alongside a first real tool so it's not abstraction without a use.

- **`platform/tool/ToolResult.java`** — success/payload/error, same shape as `AgentResult` but kept
  separate since a tool call and an agent's `handle()` are different concepts.
- **`platform/tool/ToolInvoker.java`** — runs a `Callable<T>` on its own virtual thread (the
  "sandbox" from `ARCHITECTURE.md`), with a per-call timeout (default 5s) and retry-on-timeout only
  — exponential backoff + full jitter, max 3 attempts by default. An exception the tool itself
  throws fails immediately, no retry: unlike an LLM rate limit, a tool's own exception is a logic
  error, and retrying it won't fix it. `ARCHITECTURE.md` also mentions "resource caps" beyond the
  timeout; not built — no concrete tool needs them yet, and a cap picked without a real number
  behind it would be a guess.
- **`platform/tool/ToolRegistry.java`** — scans the Spring context for `@Tool`-annotated beans at
  startup and catalogs them by name (`AnnotationUtils.findAnnotation`, not
  `bean.getClass().getAnnotation`, so a proxied bean's annotation isn't silently missed). Logs each
  registration — confirmed for real on app boot, see Verified below.
- **`platform/retry/JitteredExponentialBackoff.java`** — the backoff-with-jitter math extracted out of
  `LlmClient`, now shared with `ToolInvoker`. This is the first time this codebase has generalized
  something *after* a second real user showed up rather than before — same discipline that's kept
  `EventConsumer` and `ToolRegistry` itself unbuilt until there was a concrete second case, just
  pointed at extraction instead of restraint this time.
- **First real tool: `verticals/triage/tools/SearchKbTool.java`**, backed by a hardcoded
  `KnowledgeBase` (`verticals/triage/kb/`, 5 chunks covering billing/bugs/features/abuse). Scoring
  is naive keyword overlap — explicitly not the hybrid BM25 + pgvector retrieval
  `ARCHITECTURE.md` describes. It's the simplest thing that returns relevant chunks for a query
  today; swapping in real BM25 or embeddings later doesn't change how the tool is registered or
  invoked.

## Why no `RetrieverAgent`/second Kafka consumer today

That was the other next-session candidate, and it's still the more natural place to actually *use*
`searchKb`. But building agent + Kafka consumer + tool registry in the same session risks getting
all three half-right instead of one fully verified. Today's registry and tool are real and tested
standalone; wiring them into a Retriever step that consumes `hivemind.triage.classified` and
publishes to `hivemind.triage.retrieved` is now a much smaller, better-scoped next session — and
it'll also be the second real Kafka consumer needed before `EventConsumer` is worth generalizing.

## Verified

- `./mvnw test` — **19 tests green**, up from 10 (`JitteredExponentialBackoffTest`, 4× `ToolInvokerTest`
  covering first-try success / retry-then-succeed / retry-exhaustion / no-retry-on-logic-error, 3×
  `SearchKbToolTest`, and `ToolRegistrationIntegrationTest` — a real `@SpringBootTest`, not mocks,
  proving `SearchKbTool` is actually discovered by `ToolRegistry` and actually callable through
  `ToolInvoker` in a live Spring context).
- Started the real app (`ANTHROPIC_API_KEY=dummy ./mvnw spring-boot:run`) and grepped the boot log:
  `Registered tool 'searchKb' (vertical=triage) -> SearchKbTool` — confirms the registration
  actually runs in the full application context on startup, not only under test properties. Shut
  down cleanly afterward.
- No new HTTP surface today (nothing calls `searchKb` from the API yet), so no curl step — the
  `@SpringBootTest` above is the real end-to-end proof for this slice, the same way it would be for
  any pure in-process wiring with no external broker/API involved.

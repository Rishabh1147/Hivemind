# Devlog — 2026-07-21

## What got built

The `RetrieverAgent` — the immediate next-session recommendation from 2026-07-20's devlog. This is
the first time two agents genuinely coordinate through the event bus: classify feeds retrieve.

- **`verticals/triage/agents/RetrieverAgent.java`** — `@AgentRole(vertical="triage",
  role="retriever")`. Looks up `searchKb` by name from `ToolRegistry` (not injected directly as
  `SearchKbTool`) and calls it through `ToolInvoker`, the same path any future planner-dispatched
  agent would use. Configurable `top-k` via `hivemind.triage.retriever.top-k` (default 3).
- **`verticals/triage/messaging/TicketClassifiedConsumer.java`** — the second real Kafka consumer.
  `@KafkaListener` on `hivemind.triage.classified`; skips retrieval for tickets whose classification
  failed (nothing meaningful to search for), otherwise runs `RetrieverAgent` and publishes a
  `TicketRetrieved` event to the new `hivemind.triage.retrieved` topic.
- **Split `TicketClassified` out of `TriageResponse`.** `ClassifyRequestConsumer` used to publish
  `TriageResponse` directly to `hivemind.triage.classified` — fine while nothing consumed it, but
  `RetrieverAgent` needs the original ticket body to search against, and `TriageResponse` (the
  HTTP-facing DTO) never carried it. Rather than bolt a `body` field onto the API response type,
  added `verticals/triage/events/TicketClassified.java` as the dedicated Kafka payload
  (`ticketId, ticketBody, status, category, confidence, error`). `TicketStatusStore` still stores
  `TriageResponse` for the GET endpoint; the two types now diverge on purpose.
- **`TriageContextKeys.java`** — the `"ticketBody"` context key, extracted out of `ClassifierAgent`
  (where it lived as `TICKET_BODY_KEY`) now that `RetrieverAgent` is a second real reader of it. Same
  discipline as `JitteredExponentialBackoff`'s extraction on 2026-07-20 — wait for a second user,
  then extract, applied again today rather than just written about.
- **`hivemind.triage.retrieved` topic** — `TriageTopics.RETRIEVED`, declared as a `NewTopic` bean in
  `KafkaConfig` alongside the other two.

## Why retrieval doesn't show up in `GET /tickets/{id}` yet

`TicketClassifiedConsumer` deliberately does not write into `TicketStatusStore`. There's no
Responder yet to consume retrieved chunks into a user-facing result, and bolting a `chunks` field
onto `TriageResponse` before something actually renders them would be speculative. The event log is
still the source of truth for this stage — verified directly against `hivemind.triage.retrieved`
today (see Verified), the same way `hivemind.triage.classified` was verified in isolation before
anything consumed it on 2026-07-17.

## Verified

- `./mvnw test` — **21 tests green**, up from 19: 2 new `RetrieverAgentTest` cases (matches found via
  the real `SearchKbTool`/`KnowledgeBase`, and a graceful failure when the tool isn't registered),
  and `TriageKafkaIntegrationTest` extended to run a ticket through **both** Kafka stages against a
  real Testcontainers broker — POST → `hivemind.triage.classify` → `hivemind.triage.classified` →
  `hivemind.triage.retrieved`, reading the final event directly off the topic with a raw consumer
  (`KafkaTestUtils`) rather than only checking the HTTP-visible status. `ClassifierAgent` is the only
  mock in that test — `RetrieverAgent`, `ToolRegistry`, `ToolInvoker`, and `SearchKbTool` all run for
  real, since none of them touch anything external or costly.
  - Caught and fixed a real bug while writing this: `KafkaTestUtils.consumerProps(...)` takes
    `(brokerAddresses, group, autoCommit)`, not `(group, autoCommit, brokerAddresses)` — passed them
    in the wrong order first, which surfaced as `ConfigException: Invalid value PLAINTEXT://... for
    enable.auto.commit`. Fixed by checking the actual parameter order rather than guessing again.
- Manual verification against `docker-compose`'s real broker, deliberately isolating the new code
  from the classify stage rather than needing a real Anthropic key: started the real app, then used
  `kafka-console-producer.sh` to publish a hand-built `TicketClassified` JSON message directly to
  `hivemind.triage.classified` (`status: "classified"`, body `"I was charged twice for my
  subscription"`). `kafka-console-consumer.sh` on `hivemind.triage.retrieved` showed the real
  result: `billing-duplicate-charge` ranked first, `billing-failed-payment` second — correct,
  real `SearchKbTool` scoring, not a fixture.
  - Then published a second message with `status: "classification_failed"` and confirmed **no**
    message appeared for it on the retrieved topic, and the app log showed exactly why:
    `Skipping retrieval for ticket manual-test-2 (classification status: classification_failed)`.
  - Torn down cleanly afterward (`docker compose down`, app process killed).

## Resume bullet status

- *"agents that plan, dispatch, and coordinate subtasks via a Kafka event bus"* — **"coordinate
  subtasks" is now honestly partially true**: classify and retrieve are two agents genuinely
  chained through Kafka, each independently consumable/scalable. Still not true: no planner exists
  to *decide* that chaining dynamically — the classify→retrieve link is hardcoded in
  `TicketClassifiedConsumer`, not planner-routed.
- Tool registry (2026-07-20) is now actually used in the live pipeline, closing yesterday's
  explicitly-flagged gap ("nothing in the running system currently calls a tool during a request").

# Devlog — 2026-07-17

## What got built

The Kafka event bus — the top candidate flagged in both prior devlogs, unblocked today because
Docker's daemon is confirmed running. Ticket classification is no longer a synchronous in-process
call; it's dispatched over a real broker.

- **`docker-compose.yml`** — single-broker Kafka in KRaft mode (`apache/kafka:3.8.0`, no
  ZooKeeper), health-checked via `kafka-broker-api-versions.sh`.
- **`platform/messaging/EventBus.java`** and **`TopicNaming.java`** — the vertical-agnostic
  messaging core. `EventBus` wraps Spring Kafka's `KafkaTemplate`, serializing payloads to JSON
  with Jackson — same indirection rationale as `LlmClient` for Claude: every vertical publishes
  through one seam, so OTel span injection and outbox buffering (both on the roadmap) get added
  once. `TopicNaming.of(vertical, stage)` enforces the `hivemind.<vertical>.<stage>` convention
  from `ARCHITECTURE.md`.
- **`verticals/triage/messaging/TriageTopics.java`** — `CLASSIFY` / `CLASSIFIED` topic name
  constants. These are string literals, not `TopicNaming.of(...)` calls, because
  `@KafkaListener(topics = ...)` requires a compile-time constant expression and a static method
  call doesn't qualify. `TopicNamingTest` asserts the literals match the convention instead, so the
  two can't silently drift.
- **`infra/config/KafkaConfig.java`** — declares both topics as `NewTopic` beans (3 partitions, RF
  1) instead of relying on broker auto-create, which production clusters typically disable.
- **`verticals/triage/messaging/ClassifyRequestConsumer.java`** — `@KafkaListener` on
  `hivemind.triage.classify`. Deserializes a `ClassifyRequested` event, runs the existing
  `ClassifierAgent` unchanged, publishes the resulting `TriageResponse` to
  `hivemind.triage.classified`, and writes it into `TicketStatusStore`.
- **`TicketStatusStore.java`** — in-memory `ConcurrentHashMap`, the read side for ticket status
  now that classification happens off the request thread. Explicitly a stand-in for the
  Postgres-backed `AuditLog` in the roadmap, same pattern as `AgentContext`'s existing
  "stand-in until Kafka replaces this" comment.
- **`TriageController`** rewired: `POST /api/v1/triage/tickets` now publishes a `ClassifyRequested`
  event and returns `202 Accepted` with a `pending` status immediately, instead of blocking on the
  LLM call. New `GET /api/v1/triage/tickets/{id}` reads current status from `TicketStatusStore`
  (`404` if the ID is unknown).

## Why 202+poll instead of keeping the synchronous 200

This is the actual architectural tradeoff Kafka-between-agents buys, made concrete: the request
thread no longer waits on Claude. `ARCHITECTURE.md`'s lifecycle has a `Stream` step (SSE to a
dashboard) as the intended way to watch a ticket move through stages — that's not built yet, so
polling `GET /tickets/{id}` is the interim substitute. Swapping SSE in later doesn't change the
event-bus wiring, just the read side.

## Why no generic `EventConsumer` base class yet

`PROJECT_STRUCTURE.md` sketches one (`platform/messaging/EventConsumer.java`, "Kafka consumer
base"), but there's only one consumer today. Extracting the deserialize/error-contain shape from a
single example is guessing at a shape, not generalizing one — same discipline already applied to
not building `ToolRegistry` before a second tool exists. Revisit once a second consumer (Retriever,
Responder) needs the same pattern.

## A real environment bug, not a code bug

`mvn test` initially failed only on the new Testcontainers-backed Kafka test, with docker-java
sending API version `1.32` to a daemon whose `MinAPIVersion` is `1.40` — a stale default, not a
real negotiation result (confirmed the daemon's actual `/version` response separately via curl:
`ApiVersion: 1.55`, `MinAPIVersion: 1.40`). Traced it to the exact docker-java build shaded into
`testcontainers:1.20.4` (its `RemoteApiVersion` enum tops out at `1.44` and doesn't reliably
negotiate against this Docker Desktop build). Bumping `testcontainers.version` to `1.21.4` fixed it
outright — no workaround needed once on a version with correct negotiation. Worth remembering for
an interview: a "the test framework is broken" instinct is usually wrong; here it genuinely was,
and the fix was a dependency bump after actually reading the shaded class file to confirm the
version ceiling, not a guess.

## Verified

- `./mvnw test` — **10 tests green**, up from 8: the pre-existing 8 unchanged, plus
  `TopicNamingTest` (naming-convention assertion) and `TriageKafkaIntegrationTest` — a
  Testcontainers-backed test that runs a ticket through a **real** Kafka broker end to end
  (`ClassifierAgent` mocked deliberately, since this test is about the event-bus plumbing, not
  re-verifying classification logic already covered by `ClassifierAgentTest`).
- Manual verification against `docker-compose`, not just the ephemeral test broker: started the
  broker, ran the actual Spring Boot app, and curled it for real.
  - `POST /api/v1/triage/tickets` → `202 Accepted`, `{"status":"pending", ...}` immediately.
  - Polling `GET /api/v1/triage/tickets/{id}` a moment later returned
    `{"status":"classification_failed", "error":"Classifier LLM call failed: ... invalid x-api-key"}`
    — the classifier really did run inside `ClassifyRequestConsumer`, called the real Anthropic
    API with a deliberately bad key, and the failure round-tripped back through Kafka into the
    status store. Same fast-fail behavior as the 2026-07-15/16 sessions, now proven to survive
    going through a broker instead of an in-process call.
  - `GET /api/v1/triage/tickets/does-not-exist` → `404`.
  - `kafka-topics.sh --describe` confirmed both topics exist with 3 partitions / RF 1, matching
    `KafkaConfig`'s `NewTopic` beans exactly — the explicit-declaration path actually ran, not just
    compiled.
  - Torn down cleanly afterward (`docker compose down`, app process killed) — nothing left running.

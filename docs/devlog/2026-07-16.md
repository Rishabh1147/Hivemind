# Devlog — 2026-07-16

## What got built

Closed the gap flagged in yesterday's devlog: `LlmClient` now retries transient Claude failures
instead of treating every failure identically.

- **`platform/llm/LlmClient.java:1`** — `complete()` now retries on `dev.langchain4j.exception.RetriableException`
  (rate limits, 5xx/upstream errors) with exponential backoff + full jitter, up to a configurable
  `maxAttempts` (default 3, `hivemind.llm.retry.max-attempts` / `.base-backoff-ms` in
  `application.yml`). Anything that isn't retriable (auth errors, malformed requests) fails
  immediately with zero retry delay.
- Extracted the actual provider call into a `protected doChat(...)` method, separate from the
  retry loop in `complete()`. This wasn't originally planned — it came out of a test-design
  problem (see below) — but it's also a cleaner seam: the retry *policy* doesn't need to know
  anything about LangChain4j's response shape, only whether the call succeeded or threw.

## Why LangChain4j made this easy: it already ships the classification

Went looking for how to tell "worth retrying" apart from "don't bother," expecting to have to
define that distinction myself. LangChain4j already has it: `RetriableException` (parent of
`RateLimitException`, `InternalServerException`, `TimeoutException`) vs. `NonRetriableException`
(parent of `AuthenticationException`, `InvalidRequestException`, `ModelNotFoundException`). So the
retry logic is just `catch (RetriableException e)` — anything else propagates untouched. Confirmed
this by reading the actual exception source on GitHub rather than assuming a shape from memory.

## A real test-design problem, not a bug this time

First attempt at `LlmClientTest` mocked `ChatModel`/`ChatResponse`/`AiMessage` with Mockito, the
same way `ClassifierAgentTest` mocks `LlmClient` a layer up. It didn't work —
`when(response.aiMessage())` blew up with `UnfinishedStubbingException`, which is Mockito's way of
saying it couldn't intercept the call and a *real* method ran instead. `ChatResponse` is a
LangChain4j-internal type not designed to be mocked (likely final/record-shaped) — fighting the
SDK's response type to make it mockable would have been the wrong fix.

Instead: extracted the one line that actually touches `ChatModel`/`ChatResponse` into
`protected doChat(...)`, and tested the retry loop with a small test subclass
(`ScriptedLlmClient`) that overrides `doChat` to throw or return scripted values — no LangChain4j
types involved in the test at all. This is a case where the test being awkward to write was a
signal about the code's seams, not the test's fault.
